## Reference repo ground truth (from `_reference/resharper-mcp/CLAUDE.md` and source)

Superseded/confirmed details vs. earlier web research:

- Reference targets `JetBrains.ReSharper.SDK` **2025.3.3** (Rider 2025.3), net472, `LangVersion=latest`. Only two PackageReferences: `JetBrains.ReSharper.SDK` and `Newtonsoft.Json 13.0.3`.
- **No `[ZoneMarker]` class exists anywhere in the reference repo.** The plugin works without one — it doesn't restrict itself to a Rider-only zone. For our VS port: start M0 the same way (no explicit ZoneMarker) since that's the proven-working baseline; only add `IRequire<IVisualStudioZone>` later if we actually observe the plugin being offered/loaded somewhere undesired, rather than adding unproven ceremony up front.
- `[SolutionComponent]` requires the non-obsolete constructor form in current SDKs: `[SolutionComponent(Instantiation.ContainerAsyncAnyThreadSafe)]` (enum lives in `JetBrains.Application.Parts`). Applies equally to `[ShellComponent]`.
- PSI operations cannot run on thread-pool threads — error message is literally *"This action cannot be executed on the .NET TP Worker thread"*. Fix: inject `IShellLocks`, dispatch via `ExecuteOrQueueReadLock(name, action)`; block the HTTP thread on a `ManualResetEventSlim` (reference uses a 30s timeout in prose but the code's actual constant — verify when porting `McpServerComponent`).
- `IPsiSourceFile.GetPrimaryPsiFile()` does not exist — use `sourceFile.GetPsiFiles<KnownLanguage>()` (language-agnostic) or `GetDominantPsiFile<CSharpLanguage>()`.
- `DocumentCoords` needs typed intrinsics: `(Int32<DocLine>)(line - 1)`, `(Int32<DocColumn>)(column - 1)`.
- `GetDocumentRange()` is an extension in `JetBrains.ReSharper.Psi.Tree.TreeNodeExtensions` — call explicitly if overload resolution fails.
- `IDocument.GetCoordsByOffset()` is obsolete — use `documentOffset.ToDocumentCoords()`.
- `PresentationLanguageForTests` doesn't exist — fall back to `CSharpLanguage.Instance`.
- `ITypeElement.IsAbstract()` doesn't exist — cast to `IModifiersOwner` and read `.IsAbstract`.
- `GetContainingNamespace()` lives on `ITypeElement`, not `IClrDeclaredElement` — for non-type elements go via `GetContainingType().GetContainingNamespace()`.
- `LibrarySymbolScope` lives in `JetBrains.ReSharper.Psi.Caches`.
- `GetProjectReferences` requires iterating `project.TargetFrameworkIds` and calling per-TFM; `IProjectToProjectReference` has no `ResolveReferencedProject()` — use `GetReferencedName()`.
- No public "get current highlightings" API on `IDaemon` — file-error tools walk the PSI tree directly (`IErrorElement` nodes, `reference.Resolve().ResolveErrorType`).
- Reference's own claim: *"The official C# MCP SDK (ModelContextProtocol NuGet) requires .NET 8+ for HTTP transport, but ReSharper plugins target net472."* — **this is the claim Step 1's spike exists to re-verify**, since `ModelContextProtocol.Core` (as opposed to `ModelContextProtocol.AspNetCore`) turns out to target `netstandard2.0` too. See spike results below once run.
- Confirmed by reading `McpServerComponent.cs` directly (not just CLAUDE.md prose, which says 30s and is wrong/stale): the tool-call timeout is `private const int ToolTimeoutSeconds = 120;`, used as `done.Wait(TimeSpan.FromSeconds(ToolTimeoutSeconds))`. Our port should use 120s, matching the plan.

## Step 1: MCP transport spike — RESULT: adopt the official SDK

Ran the spike in `spikes/McpTransportSpike/` (standalone net472 console app, not part of the shipped plugin — kept for reference). **Decision: use `ModelContextProtocol.Core` instead of hand-rolling JSON-RPC/MCP**, overturning the reference repo's approach. Concretely proven, not just theorized:

- `ModelContextProtocol.Core` 1.4.1 restores and builds cleanly against `net472` (via its `netstandard2.0` TFM) with zero code changes needed — `dotnet build` succeeded on the first attempt referencing it from an `OutputType=Exe`, `TargetFramework=net472` csproj.
- Wired `StreamableHttpServerTransport` (ctor takes just `ILoggerFactory`) directly to a plain `System.Net.HttpListener`: POST bodies are deserialized with `JsonSerializer.Deserialize<JsonRpcMessage>(stream, McpJsonUtilities.DefaultOptions)` then passed to `transport.HandlePostRequestAsync(message, response.OutputStream, token)`, which writes the SSE-framed JSON-RPC response directly to the given stream. GET requests go to `transport.HandleGetRequestAsync(response.OutputStream, token)`.
- `McpServer.Create(transport, options, loggerFactory, serviceProvider)` + `server.RunAsync(token)` (started as a background task) is the whole server bootstrap. Tools are registered via `options.ToolCollection.Add(McpServerTool.Create(delegate, createOptions))` — **and the JSON input schema is auto-inferred from the C# delegate's parameter types**, not hand-written. This is a real reduction in porting work for `Tools/*.cs`, not just less protocol code.
- Full round trip verified against the running exe with real `curl` requests: `initialize` → correct MCP handshake response with `protocolVersion`/`capabilities`/`serverInfo`; `notifications/initialized` → silently accepted; `tools/list` → returned the registered tool with correct auto-generated schema; `tools/call` → executed the delegate and returned a correct `CallToolResult` content block. GET (SSE) opened without erroring.
- **Gotcha found, not a blocker**: `StreamableHttpServerTransport` does not itself set the `Mcp-Session-Id` response header — the ASP.NET Core-hosted package presumably does that in its own middleware. Our `McpHttpServer` (M1) needs to read `transport.SessionId` after the `initialize` response and set that header manually, matching the MCP Streamable HTTP spec. Not different in kind from what the original hand-rolled plan already had to do.
- **Real dependency chain now known exactly** (see `bin/Debug/net472/` after building the spike): `Microsoft.Bcl.AsyncInterfaces`, `Microsoft.Bcl.Memory`, `Microsoft.Extensions.AI.Abstractions`, `Microsoft.Extensions.DependencyInjection.Abstractions`, `Microsoft.Extensions.Logging.Abstractions`, `System.Buffers`, `System.Diagnostics.DiagnosticSource`, `System.IO.Pipelines`, `System.Memory`, `System.Net.ServerSentEvents`, `System.Numerics.Vectors`, `System.Runtime.CompilerServices.Unsafe`, `System.Text.Encodings.Web`, `System.Text.Json`, `System.Threading.Channels`, `System.Threading.Tasks.Extensions`. This is the exact list that must coexist with whatever devenv.exe/ReSharper already has loaded — **still unverified inside the actual host process**, which is exactly what M1's "re-verify inside devenv.exe" step exists to check. The standalone spike could not surface host-process assembly conflicts by design.
- Upstream issue `csharp-sdk#592` ("Streamable HTTP server support under .NET Framework") is now answered empirically for our purposes even though it's still open/unconfirmed upstream: **it does work**, at least standalone. Worth commenting on that issue once M1 confirms it also works inside devenv.exe, since nobody else has published this.

**Net effect on the plan**: M1 now targets the official-SDK path (Section 2's "port verbatim" plan for `Protocol/JsonRpc.cs`/`Protocol/McpTypes.cs` is dropped — those become unnecessary, replaced by direct use of `ModelContextProtocol.Core` types). The hand-rolled fallback remains fully documented in the plan if M1's in-devenv.exe re-check surfaces a blocking assembly conflict, but is no longer the default path.

## M0: concrete evidence of the assembly-conflict risk (found during `dotnet restore`, not yet at runtime)

Initially added `ModelContextProtocol.Core` to `ReshaperVsMcp.csproj` in M0 by mistake (scope creep — the plan puts it in M1). Restoring with **both** `JetBrains.ReSharper.SDK 2026.1.4` and `ModelContextProtocol.Core 1.4.1` referenced together produced ~35 `NU1608` warnings, all of the same shape:

> `JetBrains.Platform.*` 261.0.20260701.132435 requires version `System.Memory` (= 4.6.0), but version `System.Memory` 4.6.3 was resolved.

Same pattern for `System.Threading.Tasks.Extensions` (4.5.4 wanted vs 4.6.3 resolved), `System.Threading.Channels` (8.0.0 vs 10.0.7), `Microsoft.Bcl.AsyncInterfaces` (8.0.0 vs 10.0.7), `System.Buffers` (4.6.0 vs 4.6.1), `System.Diagnostics.DiagnosticSource` (8.0.1 vs 10.0.7), `Microsoft.Extensions.DependencyInjection.Abstractions`/`Microsoft.Extensions.Logging.Abstractions` (8.0.0 vs 10.0.7), `System.Runtime.CompilerServices.Unsafe` (6.1.0 vs 6.1.2), `System.ValueTuple` (4.5.0 vs 4.6.2).

**What this means**: `JetBrains.ReSharper.SDK` pins these shared BCL-polyfill assemblies to *exact* versions — almost certainly matching what's physically bundled with the installed ReSharper 2026.1.4 (Wave 261) and therefore already loaded inside `devenv.exe`. `ModelContextProtocol.Core`'s own dependency floor is higher for several of them. NuGet's default resolution silently took the higher versions (a warning, not a build error) for our project's own build — but that only guarantees a self-consistent *build*, not a self-consistent *runtime* once loaded alongside ReSharper's own already-resident copies of the lower versions inside the same `devenv.exe` AppDomain.

This is not yet a confirmed failure — .NET Framework *can* load two differently-versioned copies of an assembly with the same simple name side by side in one AppDomain in some circumstances, and it may simply work if our code never passes an MCP-SDK type that closes over one of these BCL types across a ReSharper-API boundary. But it is a concrete, evidenced reason to treat M1's "re-verify inside devenv.exe" step as load-bearing, not a formality. Two mitigation options to try in M1, in order:

1. **Pin the conflicting transitive versions down** in `ReshaperVsMcp.csproj` to match the exact versions `JetBrains.Platform.*` wants (`System.Memory 4.6.0`, `System.Threading.Tasks.Extensions 4.5.4`, `System.Threading.Channels 8.0.0`, `Microsoft.Bcl.AsyncInterfaces 8.0.0`, `System.Buffers 4.6.0`, `System.Diagnostics.DiagnosticSource 8.0.1`, `Microsoft.Extensions.DependencyInjection.Abstractions 8.0.0`, `Microsoft.Extensions.Logging.Abstractions 8.0.0`, `System.Runtime.CompilerServices.Unsafe 6.1.0`, `System.ValueTuple 4.5.0`) via explicit `<PackageReference>` entries, and confirm `ModelContextProtocol.Core` still functions correctly against those older floors (its own nuspec likely declares `>=` minimums, not exact pins, so downgrading to the SDK's exact versions should be compatible — needs an actual re-run of the M1 spike-equivalent to confirm).
2. If pinning down doesn't fully resolve it (e.g. `ModelContextProtocol.Core` hard-requires a feature only in the newer version), fall back to the hand-rolled JSON-RPC approach per the original plan.

Fixed for now by **not** referencing `ModelContextProtocol.Core` from `ReshaperVsMcp.csproj` in M0 — M0 stays scoped to proving the bare `JetBrains.ReSharper.SDK`-only plugin loads, isolating that variable before M1 reintroduces the MCP SDK and this conflict has to be actually resolved.

## M0: first install attempt — three real findings, all fixed

First attempt failed before the plugin ever loaded, so nothing below invalidates the code itself — all three are packaging/verification-method issues, now fixed:

1. **Package ID needs a dot-delimited "company" prefix.** Install failed with: *"Could not generate JetMetadata for a third-party package ReshaperVsMcp/0.0.1-m0. The product package identifier is expected to start with the company name, delimited with a dot."* ReSharper's Extension Manager metadata generator requires IDs shaped like `Company.Product` (matching what the earlier research already showed for JetBrains's own samples — `MyCompany.SampleReSharperPlugin` — but I didn't carry that requirement into our nuspec). **Fixed**: `<id>` changed from `ReshaperVsMcp` to `Local.ReshaperVsMcp` in `ReshaperVsMcp.nuspec`. `Local` is a placeholder — needs a real decision before any actual publish in M5 (e.g. a real company/handle name), but unblocks local dev now.
2. **Prerelease versions are hidden by default in Extension Manager.** Our version `0.0.1-m0` required manually ticking "include pre-release packages" in the source list to even show up in search — easy to miss, and not obvious the package existed at all otherwise. **Fixed**: dropped the prerelease suffix, version is now plain `0.0.1`. Milestone-tagging via version suffix isn't worth the friction at this stage; will revisit real semantic versioning around M5.
3. **`logger.Info(...)` alone is not a reliable verification signal.** No log lines for our plugin showed up at all (only the JetMetadata error, which makes sense in isolation — the package didn't actually install due to finding #1, so nothing had a chance to load and log). But independent of that: ReSharper's default log verbosity can suppress `Info`-level messages, so even a successful load might not have produced a visible line — the log-grep approach was fragile as a primary signal from the start. **Fixed**: `McpShellComponent` now also writes a marker file to `%TEMP%\ReshaperVsMcp\shell-component-loaded.txt` (timestamped) on construction, wrapped in try/catch so a write failure can't crash the component. This is the new **primary** verification signal — unambiguous, log-level-independent, one `Test-Path`/`Get-Content` away. The `logger.Info` call stays as a secondary signal for later, deeper debugging.

## M0: RESULT — PASS (confirmed 2026-07-10)

## M1: assembly-pinning attempt — tried, hard-fails, reverted

Tried mitigation option 1 from the M0 section above (pin the transitive BCL versions down to `JetBrains.Platform.*`'s exact pins). **Does not work**: NuGet's downgrade protection turns it into hard `NU1605` errors, not warnings — `ModelContextProtocol.Core`'s own dependency chain has a genuine *minimum* version floor (e.g. requires `Microsoft.Extensions.Logging.Abstractions >= 10.0.7`, `System.Memory` transitively >= 4.6.3 via `Microsoft.Bcl.Memory`) that is *higher* than what `JetBrains.Platform.*` pins to. ## M1: retry made it worse — ReSharper stopped loading in VS entirely

This is the important one. The retry install did NOT just fail to install our plugin — it broke ReSharper's own ability to load in VS at all (confirmed by the user directly: "crashed ReSharper to the point it no longer loads"). Full incident + recovery below, since the root cause changes how this project ships dependencies going forward.

**Root cause (best-evidenced theory)**: 0.0.2's nuspec shipped the *full* resolved dependency closure for `ModelContextProtocol.Core` — 16 DLLs, including 12 that are also part of ReSharper's own platform install (confirmed directly by diffing against a live `ReSharperPlatformVs18_959835b5` install directory: `Microsoft.Bcl.AsyncInterfaces`, `Microsoft.Extensions.DependencyInjection.Abstractions`, `Microsoft.Extensions.Logging.Abstractions`, `System.Buffers`, `System.Diagnostics.DiagnosticSource`, `System.IO.Pipelines`, `System.Memory`, `System.Numerics.Vectors`, `System.Runtime.CompilerServices.Unsafe`, `System.Text.Encodings.Web`, `System.Text.Json`, `System.Threading.Channels`, `System.Threading.Tasks.Extensions` were all already present). The install log showed Extension Manager's conflict resolver picking *our* (newer) version at each of these shared extraction paths ("Choosing X (in Local.ReshaperVsMcp/0.0.2)"). That's not neutral — those are almost certainly **shared physical files** that ReSharper's own core code also loads. Overwriting them with different, newer versions likely broke ReSharper's own strong-name-bound references to its originally-shipped exact versions, which would explain total ReSharper failure (not just our plugin failing) far better than "our plugin alone had a bad dependency."

**Recovery steps taken** (all with the machine directly reachable from this session, same user/filesystem):
1. Force-closed the stuck `devenv.exe` (user-authorized).
2. Deleted the broken `ReSharperPlatformVs18_959835b5_001` hive folder (user-authorized) — confirmed safe first: base platform files are cached separately in a sibling `Installations\Packages\` folder (untouched `.nupkg` cache), and no other config outside the hive itself referenced our plugin (checked `%APPDATA%\JetBrains\` — nothing; the "installed extensions" list lives in `ExtensionVerificationReport.json` *inside* the hive, confirmed by reading it directly — deleting the hive genuinely removes all trace, no risk of an outside config re-triggering our broken plugin's reinstall).
3. Relaunching `devenv.exe` alone did **not** trigger ReSharper to self-heal — no new hive folder, no new logs, meaning VS itself loads fine but ReSharper's own hook doesn't auto-repair on launch.

## M4: runtime verification, round 1 — `generate_members` PASS, `rename_symbol` FAIL (silent)

`0.0.9` installed cleanly (user confirmed, no repeat of the earlier host-corruption saga). Tested against a real file, `D:\src\RealWorldSolution\RealWorldSolution.ServiceModel\Types\PriceEntryDto.cs` (disposable branch, user explicitly authorized "go crazy" write-tool experimentation), reverted to original content after each test:

- **`generate_members` (`constructor` kind)**: PASS. Emitted a correct constructor assigning all 7 properties, diffed against the file before/after.

- **`rename_symbol` (dry-run)**: FAIL. `tools/call` for `rename_symbol` on `PriceEntryDto` → `PriceEntryDtoRenamed` (dryRun:true) returned only `{"isError":true,"content":[{"type":"text","text":"An error occurred invoking 'rename_symbol'."}]}` — no exception detail anywhere, including a full grep of the live ReSharper session log for `rename_symbol`. **Root cause of the silent failure**: `McpHttpServer` constructed `StreamableHttpServerTransport(null)` and `McpServer.Create(_transport, options, null, null)` — `null` for `loggerFactory` in both calls. The MCP SDK catches exceptions thrown by a tool delegate internally and reports back only the generic `"An error occurred invoking 'x'"` message over JSON-RPC (by design — don't leak internal exception details/stack traces to a remote caller), but *logs* the real exception via the injected `ILoggerFactory`. With `null` passed, that logging had nowhere to go — not suppressed by log level, structurally impossible to have been written anywhere.

**Fix**: new `Host/JetBrainsLoggerFactory.cs` bridges `Microsoft.Extensions.Logging.ILoggerFactory`/`ILogger` (what the SDK wants) onto `JetBrains.Util.ILogger` (ReSharper's own logger, already flowing into the same log file the marker-file/`logger.Info` calls use). Wired into `McpHttpServer`'s constructor in place of the two `null`s.

**Compile gotcha worth remembering**: `JetBrains.Util` and `Microsoft.Extensions.Logging` both declare a type named `ILogger`. A naive `using JetBrains.Util;` (for its `ILogger` extension methods — `.Error()`/`.Warn()`/`.Info()`) alongside `using Microsoft.Extensions.Logging;` (for the interfaces `JetBrainsLoggerFactory` needs to implement) makes every *bare* `ILogger` reference ambiguous (CS0104), even though a `using X = ...;` alias for one of them already exists — the alias doesn't suppress the bare-name collision from the other wildcard `using`. Fixed by aliasing **both** (`JetBrainsLog` for JetBrains's, `MsLogger`/`MsLoggerFactory`/`MsLoggerProvider` for Microsoft's) and never referencing either bare `ILogger` name directly. `JetBrains.Util`'s wildcard `using` stays only for the extension methods; the type itself is never used unqualified.

**`GenerateMembersTool` CS0472 fix** (compiled clean, needed for `equality-members` to build at all): the hash-code-combining loop guarded every member with `x != null ? x.GetHashCode() : 0`, but for non-nullable value-type members (`int`, `bool`) that guard is dead code — MSBuild flags `CS0472` ("expression is always true since a value of type 'int' is never equal to 'null'"). Fixed by checking `m.Type.IsValueType() && !m.Type.IsNullable()` (from `JetBrains.ReSharper.Psi.Util`) and only emitting the null-guarded form for reference types and `Nullable<T>`.

## M4: important finding — PSI document cache goes stale relative to disk, doesn't self-heal

While trying to build a fresh write-tool test target, hit two escalating symptoms with the live `0.0.9` session (`devenv.exe` idle/AFK for a while):

1. Appended a scratch class directly to `RealWorldSolution.Tests\SampleFixtureTests.cs` via a plain filesystem edit (not through VS's own editor). `list_symbols_in_file` and `get_file_errors` on that file kept reporting the **old** 4-symbol content with 0 diagnostics — no sign of the new class, no sign of the unresolved `List<int>`/`Console` references it introduced. Confirmed this wasn't just watcher debounce: polled every 5s for 90s straight, still stale the whole time.
2. Independently, `list_symbols_in_file` on `RealWorldSolution.ServiceModel\Types\PriceEntryDto.cs` — a file that had a constructor + `Equals`/`GetHashCode` added via `generate_members` **earlier in this same devenv session**, then reverted back to the plain 7-property DTO via a plain filesystem edit — still reports **12 symbols including the constructor and equality members that no longer exist on disk**. Verified directly: `cat`'d the file, it's genuinely back to the clean 7-property version; searched the whole solution tree for a second `PriceEntryDto.cs` that the tool might be resolving to instead — only one file exists. This is real PSI/disk drift, not a duplicate-file mixup.

**Conclusion**: ReSharper's PSI document cache for a given file only updates when something goes through VS's own live document/editor model (opening it in an editor, or a PSI-native mutation like a write tool's own transaction, which both mutates PSI *and* persists to disk in the same operation). A plain out-of-band filesystem write — in either direction, adding new content or reverting old content — is not picked up while `devenv.exe` sits idle, for at least 90+ seconds, and apparently indefinitely without some other trigger (regaining window focus, a manual reload, or a restart). This is a superset of the earlier "brand-new file not found in solution" question from the previous verification round — it's not glob/inclusion-specific, it's "PSI doesn't notice out-of-band disk changes to already-known files either."

**Consequence for future testing**: never hand-edit a file via a plain filesystem write and then immediately drive an MCP tool against it — the tool's write path calls `CommitAllDocuments()` first, which flushes PSI's *own* in-memory state to disk, not the reverse; if PSI's cached view is stale relative to disk, a write tool touching that file risks silently writing PSI's stale content back over whatever is actually on disk (e.g., resurrecting the reverted `PriceEntryDto.cs` constructor/equality members as a side effect of an unrelated edit). Confirmed `PriceEntryDto.cs` is currently in exactly this drifted state — **do not run write tools against it until this is resolved** (needs a devenv restart or some other explicit PSI resync, not proven fixable from outside the process). `SampleFixtureTests.cs` is safe again by coincidence: both the append and the revert were invisible to PSI, so PSI's frozen view and disk's current content happen to agree.

**Practical workaround used for the rest of this verification pass**: only drive write-tool tests through the tools themselves end-to-end (mutate via the tool, verify via the tool, revert via the tool if possible) rather than seeding "before" state with a hand edit — since a PSI-native mutation is guaranteed to keep PSI and disk in sync, this sidesteps the staleness class of bug entirely. Where a hand-revert-via-filesystem is unavoidable at the end of a test, treat that file as now-possibly-drifted for the rest of the session.

**Root cause, per the SDK binaries**: grepped identifier strings out of the Wave-261 platform assemblies directly (`jetbrains.platform.core.shell`'s `JetBrains.Platform.Shell.dll`) rather than relying on public docs, which don't cover this. Found `IFileSystemTracker`, `AdviseForFileSystemChanges`, `IFileSystemChangeDeltaVisitor` — and, decisively, a flag literally named **`SuspendFileSystemTrackerWhenInactive`**. That's a direct match for the observed symptom: the platform appears to deliberately suspend its own file-change tracking while the IDE is inactive (unfocused/AFK), not a bug in this plugin or a glob/inclusion quirk. No `ForceRescan`/`ManualSync`-shaped public API turned up in the same binary string sweep, so there doesn't appear to be a callable way to force a resync from a plugin/tool — the fix is bringing the IDE back to an active state (regaining focus) or a restart (which reparses everything from disk unconditionally, sidestepping the tracker entirely). Also checked `JetBrains.Platform.VisualStudio.Core.dll`, which layers VS's own `IVsFileChangeEx`/`AdviseFileChangeAsync` underneath — so there are two suspend-capable layers (VS's own file-change advise mechanism, and ReSharper's tracker on top of it), either of which could independently explain throttled behavior while inactive.

**Confirmed fixed by restart**: after the user installed `0.1.0` and restarted `devenv.exe`, `list_symbols_in_file` on `PriceEntryDto.cs` immediately reported the correct 8 symbols (7 properties, no leftover constructor/equality members) — PSI and disk back in sync, no further drift-related caution needed for that file going forward.

## 0.4.1 verification round - the real cause of the "500 with empty body" bug was the TEST HARNESS, not `ResolveFile` (0.4.2)

With `0.4.1` installed and restarted, retested `list_symbols_in_file(filePath=...)` immediately - same file, same tool, that 500'd all through the previous round. **This time the 500 body wasn't empty** (0.4.1's own fix working as intended) and showed the real exception: `System.Text.Json.JsonException: 's' is an invalid escapable character within a JSON string`, thrown while deserializing the incoming request, at a byte offset landing inside the `filePath` value.

**Root cause: the `curl -d '...'` JSON bodies used for testing throughout the ENTIRE previous round were malformed.** There's an extra layer of unescaping between what gets typed into a `Bash` tool call's `command` string and what the underlying shell actually receives - confirmed empirically: writing `\\` (two backslashes, intending bash to see two backslashes, which is correct single-quoted-string JSON for one literal backslash) resulted in bash receiving only `\` (one backslash) once it actually reached the file/process - verified directly with `printf '%s' '...' > file` + inspecting the written bytes with `dd`/`cat -A`. Every `curl -d` call built with double-escaped Windows paths (`D:\\src\\...`) was therefore sending invalid JSON (bare `\s`, `\R`, etc. - not valid JSON escape sequences), and `System.Text.Json` correctly rejected it. Fix for future testing: use **quadruple** backslashes (`\\\\`) per intended path separator in a `curl -d '...'` body typed into a tool call, confirmed empirically to produce correct double-backslash JSON once it reaches the actual process.

**This means the whole "every filePath-based tool is broken" finding from the previous round was itself an artifact of broken test requests, not a server bug.** The `PsiHelpers.ResolveFile` unfiltered-`GetAllProjects()` issue *may* still be real (filtering out projects with no `ProjectFileLocation` is defensively correct regardless), but it was never actually confirmed as the cause of anything - the 500s stopped the moment the test JSON was fixed, before ever re-triggering whatever `ResolveFile` was doing differently. Leaving the filter in (harmless, arguably correct either way) but downgrading the earlier "root cause found" claim to "plausible defensive fix, not confirmed against a real failure."

**A second, more consequential bug surfaced once requests were well-formed: `sync_file_from_disk` failing with a fully generic, zero-detail error.** `{"isError":true,"content":[{"text":"An error occurred invoking 'sync_file_from_disk'."}]}`, HTTP 200 (not 500) - this is the MCP SDK's own tool-invocation wrapper catching an exception thrown *inside* the tool and reducing it to that one fixed generic string, a completely different code layer from the HTTP-level 500 handler fixed in 0.4.1 (which only covers exceptions *outside* the SDK's tool dispatch, e.g. request deserialization). Went looking for ReSharper's own IDE log next (the `JetBrainsLoggerFactory` routes the SDK's internally-logged version of these exceptions there) - extensive search across `%LOCALAPPDATA%\JetBrains\ReSharperPlatformVs18\...` turned up only LevelDB cache files and FUS analytics logs, no readable text log with exception detail. Rather than keep hunting for a log file, fixed it at the source: `PsiThreadDispatcher.Execute<T>` used to `throw caught;` on any tool-body exception, which is exactly what the SDK's wrapper then reduces to the generic message. Since every one of the 26 tools' `Execute` methods returns `string` in practice, changed it to format the exception directly into the tool's own string result (`"Tool 'x' failed: " + exception.ToString()`) instead of rethrowing, for `string`-returning callers (falls back to the old rethrow behavior for any hypothetical non-string `T`, though there are none currently). ## `sync_file_from_disk` - the real bug and real fix, found immediately once 0.4.2's diagnostics landed (0.4.3)

With `0.4.2` installed, re-ran `sync_file_from_disk` against the still-stale scratch file from the previous round - 0.4.2's diagnostics fix paid off immediately, returning the real exception on the first try instead of a generic message:

```
JetBrains.Diagnostics.Assertion+AssertionException: All documents should be comitted, but some are dirty
   at JetBrains.ReSharper.Psi.Files.PsiFiles.AssertAllDocumentAreCommitted(String customText)
   at JetBrains.ReSharper.Psi.Caches.PsiCaches.Update()
   at JetBrains.ReSharper.Psi.Transactions.PsiTransactionCookie.Commit()
   at ReshaperVsMcp.Core.Tools.PsiThreadDispatcher.<>c__DisplayClass2_0`1.<ExecuteWrite>b__0()
```

Root cause, obvious once visible: `SyncFileFromDiskTool.SyncOne` calls `document.ReplaceText(...)` directly on the `IDocument`, which leaves that document "dirty" (a pending, uncommitted text change) from the PSI layer's point of view. `ExecuteWrite`'s outer `PsiTransactionCookie.CreateAutoCommitCookieWithCachesUpdate(...)` auto-commits on `Dispose()` at the end of the `using` block - and that commit path asserts that **no** document is left dirty before it updates caches. Nothing between the `ReplaceText` call and the `using` block's exit ever committed that specific change, so the assertion fired every single time, unconditionally, as soon as `textChanged` was true (i.e. whenever there was an actual edit to sync - the exact case this tool exists for). This also explains why the *first-ever* test of this tool (multiple rounds ago, informally believed "confirmed working" - a claim already corrected earlier in this log) could never actually have succeeded: the bug is in the unconditional, core code path, not an edge case.

Fix: call `_solution.GetPsiServices().Files.CommitAllDocuments()` explicitly, right after `ReplaceText`, still inside the tool's own action (i.e. before the outer transaction's auto-commit-on-dispose runs). This settles the document before `PsiTransactionCookie.Commit()` ever checks for dirty documents. Three-line change, `SyncFileFromDiskTool.cs`.

**Status: 0.4.3 packed (`dist\Local.ReshaperVsMcp.0.4.3.nupkg`), NOT yet runtime-verified.** This is a small, targeted, high-confidence fix directly addressing a fully-understood root cause (unlike the previous two rounds' broader diagnostic/defensive changes) - next round should be a fast, clean confirmation: re-run the exact same stale-then-sync repro (`StaleSyncProbe` should finally show up in `list_symbols_in_file` after the sync call), then move on to the real backlog this whole multi-round detour was blocking - `inline_variable` on a real local, the open `safe_delete` "(NOT applied)" bug, and `change_signature`'s first-ever test.

## `sync_file_from_disk` - 0.4.3's fix was correct but incomplete; the real fix is 0.4.4

With `0.4.3` installed, re-ran the exact same stale-then-sync repro - **identical exception, identical stack trace**, still `AssertionException: All documents should be comitted, but some are dirty`. 0.4.3's `CommitAllDocuments()` call right after `ReplaceText` didn't help at all.

The bug in the "fix": `SyncOne`'s very next lines, unconditionally, are:

```csharp
psiFiles.MarkAsDirty(sourceFile);
psiFiles.InvalidatePsiFilesCache(sourceFile);
```

`MarkAsDirty` **re-dirties the document** immediately after the 0.4.3 fix committed it - by design, since marking a file dirty for reparse is the entire point of this tool (see the two-staleness-layers finding earlier in this log: text-level freshness alone isn't enough, the parsed tree needs to be explicitly invalidated too). So the outer `ExecuteWrite` transaction still finds a dirty document when it tries to auto-commit-with-caches-update on `Dispose()`, just now caused by our own `MarkAsDirty` call instead of by `ReplaceText`. Same assertion, same crash, no matter where `CommitAllDocuments()` gets inserted in between - short of calling it again after `MarkAsDirty`, which would then immediately undo the mark we just asked for.

**The actual fix**: this tool's job (settle a document's text, then explicitly flag it dirty for reparse) is fundamentally incompatible with `ExecuteWrite`'s outer `PsiTransactionCookie.CreateAutoCommitCookieWithCachesUpdate` transaction, which exists for tools that make one bounded, atomic PSI mutation and want it committed cleanly - not for a tool whose whole purpose is to leave something intentionally marked dirty. Switched `sync_file_from_disk` from `PsiThreadDispatcher.ExecuteWrite` to `ExecuteSelfTransactingWrite` (the same dispatch mode `rename_symbol`/`safe_delete`/`inline_variable` use, since they manage their own transactions or need none) - this skips the outer auto-commit-cookie entirely, so there's no "all documents must be clean" assertion to violate. One-line change to the `Execute` method's dispatcher call.

## Deep dive on `safe_delete`: found and fixed the actual reflection-tooling bug, then found the real root cause (0.4.6)

Chose `safe_delete` over `inline_variable` to deep-dive first: its failure is narrower (`Initialize` already confirmed succeeding, only `Execute` misbehaves), versus `inline_variable` failing at `Initialize` itself with no data yet on why.

**First: fixed the reflection tooling, which had been silently broken for a while.** The previous round's "`GetTypes()` on `NewRefactorings.dll` stopped finding `SafeDeleteWorkflow`/`SafeDeleteRefactoring`" mystery had a real cause: the cached `dll_lookup.xml` table (built early in this project for fast `ReflectionOnlyAssemblyResolve`) has a corrupted entry for `JetBrains.Lifetimes` - its value is **multiple filesystem paths concatenated into one string** (likely from the original table-building script overwriting-vs-appending incorrectly when multiple NuGet packages ship a same-named DLL), instead of a single valid path. Every reflection call that needed `JetBrains.Lifetimes` as a dependency (looked up via `$lookup[$name]` used directly as a file path) got a garbled multi-path string passed to `ReflectionOnlyLoadFrom`, which failed, which silently dropped ~400 types transitively depending on it - including, as it turns out, `SafeDeleteWorkflow` and `SafeDeleteRefactoring` themselves. Fixed by splitting on spaces and preferring the `net472` candidate in the resolve handler. This was a real, previously-undiagnosed gap in the investigative tooling this entire project has been relying on - worth remembering for any future reflection session: if a type search comes up suspiciously empty, check `LoaderExceptions` before concluding the type doesn't exist.

**With reflection actually working, found the real bug.** `SafeDeleteWorkflow`'s real base class is `JetBrains.ReSharper.Feature.Services.Refactorings.DrivenRefactoringWorkflow` (via `DrivenRefactoringWorkflow2<SafeDeleteHelperUnsupported>`), which declares the refactoring's actual lifecycle contract: `Initialize(IDataContext)` → `PreExecute(IProgressIndicator)` → `Execute(IProgressIndicator)` → `PostExecute(IProgressIndicator)`. `SafeDeleteTool` only ever called `Initialize` then went straight to constructing `SafeDeleteRefactoring` and calling `Execute()` - **`PreExecute` was never called at all**. Given `PreExecute` is presumably where the actual usage-search/conflict-population happens (analogous to `rename_symbol`'s `dataModel.LoadBaseRenames(...)` step, which IS called there), skipping it plausibly explains everything observed: `Execute()` had nothing populated to act on, so it silently returned `false` with zero conflicts, even for the most trivial possible case.

**Fix**: added `workflow.PreExecute(NullProgressIndicator.Create())` right after the transaction opens and before constructing `SafeDeleteRefactoring`; bail out early (with whatever conflicts got extracted) if it returns `false`. Added `workflow.PostExecute(NullProgressIndicator.Create())` right after `Execute()` too, matching the full four-step lifecycle. Confirmed via reflection that `InlineVarWorkflow` shares the exact same `DrivenRefactoringWorkflow` base and also has a `PreExecute` method - but `inline_variable`'s failure happens at `Initialize` itself (returns `false` before `PreExecute` would ever run), so this specific fix does not apply there; that bug needs its own separate investigation, not attempted this round per the user's explicit "choose one, deep-dive" instruction.

**Status: 0.4.6 packed (`dist\Local.ReshaperVsMcp.0.4.6.nupkg`), NOT yet runtime-verified.** This is a well-reasoned, mechanistically-grounded fix (unlike some earlier rounds' guesses), but "well-reasoned" isn't "confirmed" - next round should re-run the exact same `TrulyUnusedHelper` repro and check for `"(applied)"` plus an actual disk diff, not just trust the return value.

After VS was killed and restarted fresh (ruling out "long-running session accumulates degraded state" as the explanation, since this was now a brand-new process), the SAME pattern reproduced immediately: `safe_delete` on `TrulyUnusedHelper` threw a real, informative exception (0.4.9's `GetFirstPageOrNull` fix correctly surfaced it instead of the earlier round's generic messages) - `System.NullReferenceException` deep in `ModuleReferenceResolveContextExtensions.GetRuntimeFeatures`, called via `OverridableMemberImpl.GetRootSuperMembers` → `DeleteHierarchyMemberData.GetOptionsPage` → `SafeDeleteWorkflow.GetFirstPageOrNull`. The call completed (no hang) and returned a clean error response. But the **very next** tool call (`apply_suggestions`, read-only `ListApplicable` mode - never even reaching its own write/transaction code) hung for 30+ seconds and timed out; a follow-up `get_solution_structure` (previously always-instant) also timed out. `Get-Process devenv` showed `Responding: True` and ~0% CPU - not a UI freeze, not a busy-spin, but a genuine stuck lock: something in the PSI locking chain was waiting on a signal that would never come.

**Diagnosis**: `DeleteHierarchyMemberData.GetOptionsPage` (reached via `GetFirstPageOrNull`) calls `FinderExtensions.FindImplementingMembers` through `IPsiServices.ParallelFinder` - a **background, multi-threaded search** - as part of computing `HasOverrides`/`HasBase` before the `NullReferenceException` fired. The exception is synchronous and gets caught fine by a `try/catch` around the `GetFirstPageOrNull` call - but if the `ParallelFinder`'s background work doesn't get torn down cleanly when the surrounding call is interrupted by an exception mid-flight, it can leave an orphaned worker holding a PSI lock forever. Every subsequent operation needing that same lock (read OR write) then queues up behind it indefinitely. This is exactly the risk the original project plan flagged as its top-listed concern from day one: *"No AppDomain isolation: an unhandled exception/deadlock on the listener/worker threads can affect all of `devenv.exe`."*

**Response: reverted, don't patch further.** Tried wrapping `GetFirstPageOrNull` in `try/catch` first (0.4.9) - insufficient, since the *synchronous* exception isn't the problem, whatever background work it kicks off before throwing is. Rather than keep experimenting live against a tool that's now confirmed capable of wedging the whole IDE, reverted `SafeDeleteTool` back to the 0.4.8 shape (`Initialize` → `PreExecute` → `Execute`, no `GetFirstPageOrNull` call at all). This keeps `safe_delete` **incomplete** (`Data` never gets populated, so `Execute()` still always returns `false`, `"(NOT applied)"`) but **confirmed safe** - the 0.4.8 shape was tested multiple times this session and never hung, only ever returned a clean (if unhelpful) result. A tool that's honestly incomplete is strictly better than one that can take the whole IDE down. Real fix for `safe_delete` deferred to a future session - candidate approaches: reflect directly into `SafeDeleteHelperUnsupported.CreateData()` to populate `Data` while skipping `GetOptionsPage`'s hierarchy walk entirely (since `CreateData` itself, per its decompiled source, does no background search - only `GetOptionsPage`'s override/base-checking does), or find whatever ambient resolve-context state an interactive session sets up that lets `GetRootSuperMembers` succeed instead of NRE-ing, and replicate it safely from a headless caller.

**`apply_suggestions` is very likely an innocent bystander, not re-investigated further.** Its own 0.4.8 transaction-wrapping fix is untouched in 0.4.10. The call that hung used `all=false`/no `inspectionIds`, which routes straight to `ListApplicable` - a read-only path that never reaches the transaction-wrapped `ExecuteAction` code at all. Far more likely explanation: `apply_suggestions` was simply the next PSI-lock-needing operation issued after `safe_delete`'s orphaned `ParallelFinder` work had already wedged things, not a bug of its own. ## M6 follow-up: `list_solutions` built, plus two features from a user idea - port pinning per solution, and a VS status bar indicator

User asked to build `list_solutions` after all (cheap given the marker-file registry already existed), then had a follow-up idea while discussing it: pin each solution to the port it was last assigned (persisted locally, read before binding), and - if feasible from a ReSharper-hosted plugin at all - a read-only VS status bar item showing the current port. All three implemented this round.

**`list_solutions`**: new `Core/Tools/ListSolutionsTool.cs`, registered directly by `McpShellComponent` (not `McpServerComponent`) since it's a process/machine-wide capability, not solution-scoped - it works even before any solution is open. Reads every `instance-*.txt` in the marker directory and formats port/PID/solution/timestamp for each, marking whichever one matches `_shellComponent.Port` as "(this instance)".

**Port pinning**: the interesting design question was *where* to store the preference. Considered ReSharper's own solution-level DotSettings mechanism (`[SettingsKey]`/`.sln.DotSettings`, what the user asked about by name) but decided against it: DotSettings are designed to be shared/checked into source control across a team, and a locally-bound port number has zero meaning to anyone else who clones the repo - using that mechanism risked either polluting the repo with a machine-specific value or needing to get the "personal/local settings layer" selection exactly right, on top of taking on an entirely new, unvalidated SDK settings API dependency right after this session already found real fragility in unfamiliar SDK internals twice (`safe_delete`'s two live hangs). Went with a plain local file store instead (`SolutionPortPreferences.cs`): one file per solution under `%LOCALAPPDATA%\ReshaperVsMcp\port-preferences\`, named by a SHA256 hash of the solution's normalized full path (paths have characters unsafe as raw filenames), holding just the last-bound port. Simple, fully understood, zero new API surface risk.

Implementing this required a real architectural change: **port binding moved from eager (at `McpShellComponent` construction, before any solution is known) to lazy (triggered by `McpServerComponent.EnsureStarted`, once a solution has actually opened)** - only once a solution is known can its remembered-port preference be read and used to bias which port gets tried first. `McpShellComponent.EnsureStarted(int? preferredPort)` is idempotent (first solution to open in a process wins; a second solution opening later in the same process just inherits whatever's already bound, but still gets its own preference file updated to that port for its own next fresh launch). `BuildCandidatePorts` tries the remembered port first (if any), then falls through to the normal `23741, 23742, ...` range, deduplicated - covers both "it's free, instant win" and "someone else has it now, fall back exactly like before" cleanly. `ToolCollection` itself is now always non-null from construction (previously it was `null` until a port bound - the old signal for "should I skip tool registration" was `_toolCollection == null`, now it's `boundPort <= 0`, checked in `McpServerComponent` right after calling `EnsureStarted`).

**Status bar indicator**: investigated feasibility via decompilation before attempting anything, given VS's *native* status bar extensibility is normally a VSPackage/MEF concern, not something a ReSharper-hosted plugin would obviously have access to. Found `JetBrains.Application.StdApplicationUI.StatusBars.IStatusBar` (in `JetBrains.Platform.Shell.dll`) - `CreateIndicator(...)` returns a persistent `IJetStatusBarIndicator` (not just transient text), added via `statusBar.Indicators.Add(lifetime, indicator)`. Confirmed this is real, already-shipping infrastructure - not speculative - by decompiling `ExceptionStatusBarIndicator`, one of ReSharper's own built-in status bar indicators, and copying its exact pattern: `CreateIndicator(default(Lifetime), logger, priority)` → `Indicators.Add(lifetime, indicator)` → `indicator.Content.Value = new RichTextAutomation(richText)` for the visible content. `McpShellComponent` now takes `IStatusBar` as a constructor parameter and shows `"R# MCP: {port}"` once a port binds successfully. Deliberately no click handler - with multiple concurrent instances there's no single sensible action a click on any one instance's indicator could take.

**Risk flag, not yet tested live**: unlike the marker-file/`list_solutions` changes (which are purely additive), this batch changes core startup behavior (lazy vs. eager port binding) and adds a **brand new constructor dependency** (`IStatusBar`) to `McpShellComponent` - the very first component this plugin constructs on load. If that DI resolution has any issue, it risks blocking the whole plugin from loading, not just one tool's execution path - a materially bigger blast radius than anything shipped so far this session (`safe_delete`'s hangs, by contrast, only broke `safe_delete` calls specifically, in an already-loaded, otherwise-working plugin). The pattern is copied from real, shipping ReSharper code, not guessed - but "copied from real code" and "verified in this codebase's actual hosting context" are different claims, and only the first one is true right now. Recommend watching VS's responsiveness closely immediately after the next install that includes this.

**Status: source changes only, NOT repackaged/version-bumped** (`list_solutions` + port pinning + status bar indicator + the earlier `safe_delete` removal and marker-file fix are all queued together). Confirmed clean build (0 errors). Tool count now 26 again (25 + `list_solutions`).

## 0.5.1 confirmed installed and STILL invisible - found the real architectural cause, deferred dispatch + MCP icon (0.5.2)

## `apply_suggestions` hang - root cause found via decompilation-only investigation, partial fix (0.5.4, list/dry-run only)

Found the real root cause without any further live testing, by tracing through the two calls that behaved differently on the exact same file moments apart: `get_diagnostics` (succeeded instantly) and `apply_suggestions`'s list mode (hung). Both call the identical `DaemonHighlightingCollector.Collect(_solution, sourceFile)`. The one concrete difference: `GetDiagnosticsTool` dispatches via `PsiThreadDispatcher.ExecuteRead` (a **read** lock); `ApplySuggestionsTool` dispatched EVERYTHING - including its read-only list/dry-run paths, which mutate nothing - via `PsiThreadDispatcher.ExecuteSelfTransactingWrite` (a **write** lock). `DaemonHighlightingCollector`'s own doc comment (written earlier this project, by an earlier version of this same session) explicitly says it "Must be called on the ReSharper main thread under a **read lock**." `ApplySuggestionsTool` was violating its own dependency's documented contract - a real, concrete, self-inflicted bug, not exotic undocumented SDK behavior this time.

**Plausible mechanism** (not independently confirmed, but consistent with everything observed): `DaemonProcessBase.DoHighlighting` almost certainly dispatches real inspection/analysis work onto ReSharper's own background worker threads (this is how ReSharper's daemon/analysis engine normally works even outside this project), and those worker threads need their own read-lock access to safely inspect PSI state. A thread already holding the **exclusive write lock** (which blocks all readers) that then calls into code needing worker threads to acquire read locks - and BLOCKS waiting for those workers to finish - creates a textbook deadlock: workers can never get their read lock while the write lock is held, and the write-lock holder can never proceed because it's waiting on those same workers. This matches the observed signature exactly (idle CPU, `Responding: True`, narrow blast radius limited to PSI-lock-needing calls - not a full IDE freeze).

**Fix, scoped to what was actually confirmed hanging**: split `ApplySuggestionsTool.Execute`'s dispatch. Read-only calls (list mode: no `inspectionIds`/`all`; and `dryRun=true`) now go through `PsiThreadDispatcher.ExecuteRead` via a new `ExecuteReadOnly` method - these never mutate anything, so a read lock is both correct and sufficient. Only real fix-application calls (`inspectionIds` or `all=true`, with `dryRun=false`) still go through `ExecuteSelfTransactingWrite` via a renamed `ExecuteApply` method, since that path genuinely needs write access for `ExecuteAction`/`PsiTransactionCookie`.

**Explicitly NOT fully fixed - `ExecuteApply` likely has the identical risk, unaddressed.** The apply-fixes loop still calls `DaemonHighlightingCollector.Collect` on every iteration (to rescan for the next occurrence after each fix) - from within the SAME write-lock context needed for the actual mutation. If the deadlock mechanism above is correct, this exact same hang risk plausibly still exists for the tool's actual primary use case (applying real fixes), just never triggered/tested live this round (only the list path was actually tested and confirmed hanging). Documented this clearly with an in-code caution comment on `ExecuteApply` and in the tool's own MCP description (now explicitly warns: list/dry-run confirmed safe, real-apply mode UNVERIFIED and possibly hang-prone, use dryRun=true first). Did not attempt to fix the apply-loop's structural issue this round (would need alternating short-lived read-lock-scan / write-lock-apply dispatches per iteration instead of one continuous write-lock-held loop) or test it live, given the demonstrated hang risk and the established "no blind live retries against a tool with a confirmed hang" policy from the `safe_delete` precedent.
